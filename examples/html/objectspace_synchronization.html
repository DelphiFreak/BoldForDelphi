ObjectSpace Synchronization
Launch Delphi-example | Launch CBuilder-example

Note!  This example demonstrates features of the ObjectSpace Synchronization Extension. This is an extension product to $(PRODUCT_BASE_NAME). A license key for the extension is required to try the example. Use the License Manager to request an evaluation key.

To run the example, you have to register a propagator. To register a propagator you need to run the binary 'BoldPropagator.exe' (found under \bin) with the command line parameter '/regserver'.

You must also register the type library BoldComConnection. Execute regdlls.bat under the bin\ directory.

This is an example on of how to synchronize several object spaces. The example will help you understand:

  • How to synchronize a BoldSystem with other BoldSystems running in different applications.
  • The ObjectSpace Synchronization Server components and their usage.

How to run

To run the examples you could either start the propagator manually (this will ensure that the propagator is not terminated when no BoldSystems are registered with it) or let the application start it automatically (in this case when all clients are closed the propagator is automatically terminated).

After having registered the propagator, start two instances of BldOwnSynchronized.exe. Now if you change an object in one instance of the application (i.e. in one BoldSystem), the object is also modified in the other BoldSystem.

Synchronize

To make an object space synchronizable we need to insert in between the BoldSystemHandle and its PersistenceHandle a small chain of interconnected components, these components are:

  • TBoldIDAdderHandle
  • TBoldSnooperHandle
  • TBoldListenerHandle
  • TBoldExternalObjectSpaceEventHandler

The Buildings & Owners example has been extended so that when several instances of the application are running (i.e. several object spaces active), all will be notified when an object is modified in any of the object spaces. Each object space is informed of all changes (in other object spaces) made to any of the loaded objects. When object space A lets say receives a notification that the object with ID 10 has been modified, the default behavior is to mark object 10 as invalid and refetch that object again. Users can write event handlers to implement the desired behavior for when an object is modified, a class is modified or a conflict occurs.
